# Algorithm reverse-engineering notes

This documents how `lib/algorithm.ts` was recalibrated against a competitor
reference (Wunderfold, https://wunderfold.art), and what's still open.

## Method

Two reference pattern PDFs (MMF and Cut & Fold) were generated by Wunderfold
from a known source image ("הסיפור" in bold Hebrew type, 1012x282px PNG).
Both PDFs encode the same book: 403 total pages, 22cm book height, 202 leaves
(pages 1,3,5...403). The PDF tables were parsed (`pdftotext -table`) into
per-leaf mark arrays and compared programmatically against a pixel-accurate
re-implementation of the sampling pipeline (column splitting, darkest-pixel
collapse, segment/toggle detection) run over the real source PNG.

## Confirmed and fixed

- **Leaf count off-by-one.** `floor((last-first)/2)` undercounts by 1;
  should be `floor((last-first)/2) + 1` (202 leaves needed, not 201).
- **No cross-leaf redistribution.** The previous "advanced MMF" model pushed
  extra segments from a crowded column onto the next blank leaf. The
  reference data disproves this: blank ("Skip") leaf positions are IDENTICAL
  between the MMF and Cut & Fold files, meaning every leaf's content comes
  only from its own native column - never a neighbor's overflow.
- **Vertical scale was wrong in kind, not just in degree.** The old code
  mapped `pixelY / imageHeightPixels * pageHeightCm` - stretching the image
  to fill the entire page height. Regression against ~500 reference points
  showed the real mapping is a `background-size: contain` fit (preserve
  aspect ratio, center vertically) into `pageWidthCm x pageHeightCm`. This
  is why `pageWidthCm` was added to `BookConfig` - it wasn't in the schema
  before, so this scaling couldn't be expressed at all.
- **`calibrateResolution` / `countSegmentsAtResolution` were dead code.**
  Never called from `generateFoldingPattern`. The resolution they searched
  for doesn't exist in the reference behavior (which samples 1 column per
  leaf, always) - removed rather than wired up.
- Direction, single-segment MMF columns, and Cut & Fold's "every toggle on
  one leaf" model were already structurally correct.

## Known limitation: multi-segment MMF column selection

When an MMF column has 2+ disconnected black runs (e.g. two strokes of a
letter passing through the same column), something has to give - only one
[top, bottom] pair fits on a leaf. The reference clearly **rotates** through
the runs top-to-bottom as leaves advance (never picks the same one forever,
never merges them into one envelope). The current implementation uses
`segments[leafIndex % segments.length]`.

This was tested against all 82 multi-segment columns in the reference file.
Roughly a dozen candidate rules were tried (global counters, run-relative
counters with various increment/reset rules, nearest-to-previous-value
continuity, connected-component "least recently shown" strand tracking).
The best rule found matches ~85% of cases exactly; the current
`leafIndex % segments.length` is simpler and generalizes better but scores
somewhat lower on this one sample. None of the tested rules reached 100% -
whatever Wunderfold does for band-merging/splitting across columns is more
subtle than a static formula recovered from output text alone.

**If this needs tightening:** re-run the comparison harness (recreate it
similarly to the analysis in this session - parse both reference PDFs with
`pdftotext -table`, load the source PNG, replicate the sampling loop) against
a few more reference images with rich multi-stroke content, and try a real
connected-component tracker across adjacent columns (match each run to its
nearest-overlap predecessor, keep a per-strand "turns since shown" counter,
always surface the strand waiting longest) rather than a static index
formula - that's the most likely direction the true rule lives in, but it
needs more reference data to pin down the exact tie-breaking.

## Known limitation: exact pixel-to-cm calibration

The `pageWidthCm` field's introduction fixes the qualitative model (contain
+ center, not stretch), but reproducing Wunderfold's numbers pixel-for-pixel
also depends on threshold/anti-aliasing handling that isn't recoverable from
PDF output alone. Against the reference image: mean error ~0.1cm, 84% of
individual marks within 0.2cm, 59% exact after rounding to 0.1cm. This is a
large improvement over the previous implementation (which was structurally
wrong and matched close to 0%), but true pixel-parity will need calibration
against Wunderfold output on live images once the app is in front of users
who can compare outputs side by side.
